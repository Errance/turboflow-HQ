## 2. Identifier System

| Field | Assigned By | Type | Description |
| --- | --- | --- | --- |
| `league_code` | TurboFlow | string, <= 32 | Fixed competition code. See section 1. |
| `sig_match_id` | SIG | string, <= 64 | Stable and unique match ID on SIG side. |
| `market_code` | TurboFlow | string, <= 64 | Fixed market code. See section 4. |
| `outcome_key` | TurboFlow | string, <= 64 | Fixed outcome key. See section 4. |
| `kickoff_at` | SIG | RFC3339 UTC | Scheduled kickoff time. |
| `at` / `at_ms` | SIG | RFC3339 / int64 ms | Data production time, used by TurboFlow to discard out-of-order old frames. |

Notes:

- The same match must keep the same `sig_match_id` during its whole lifecycle.
- Time values must use UTC.
- TurboFlow deduplicates and sorts odds frames by `(league_code, sig_match_id, market_code, at_ms)`.

## 3. Match and Market Scope

This document only defines the odds data contract between SIG and TurboFlow.

- SIG does not need to connect to the TurboFlow RFQ service.
- SIG does not need to receive user order requests.
- SIG does not need to return `provider_quote_id`.
- SIG does not need to call `/accept`.
- SIG does not need to participate in settlement or payment.

SIG only needs to continuously push football odds snapshots to TurboFlow.

## 4. Market Code and Outcome Key

### 4.1 Match Markets

| market_code | Market Name | outcome_key |
| --- | --- | --- |
| `result_1x2` | Full-time result | `home_win`, `draw`, `away_win` |
| `result_ht` | Half-time result | `home_win_ht`, `draw_ht`, `away_win_ht` |
| `btts` | Both teams to score | `yes`, `no` |
| `over_under_2_5` | Total goals over / under 2.5 | `over`, `under` |
| `over_under_1_5` | Total goals over / under 1.5 | `over`, `under` |
| `over_under_3_5` | Total goals over / under 3.5 | `over`, `under` |
| `correct_score` | Correct score | `s_<H>_<A>`, for example `s_2_1` |
| `next_goal` | Next goal | `home_win`, `away_win`, `no_goal` |

### 4.2 Series / Outright Markets

| market_code | Market Name | outcome_key |
| --- | --- | --- |
| `series_champion` | Champion | `yes`, `no` |
| `series_top_scorer` | Top scorer | `yes`, `no` |
| `series_top_4` | Top 4 | `yes`, `no` |
| `series_relegation` | Relegation | `yes`, `no` |
| `series_group_first` | Group winner | `yes`, `no` |
| `series_group_qualify` | Group qualification | `yes`, `no` |

### 4.3 Outcome Common Fields

Each odds outcome contains the following fields:

| Field | Required | Type | Description |
| --- | --- | --- | --- |
| `outcome_key` | Yes | string | Must use one of the values defined in sections 4.1 or 4.2. |
| `buy_decimal_odds` | Yes | string decimal | Buy-side decimal odds, 6 decimal places, for example `"2.150000"`. |
| `sell_decimal_odds` | Yes | string decimal | Sell-side decimal odds, 6 decimal places, for example `"1.950000"`. |
| `decimal_odds` | Optional | string decimal | Legacy single-odds fallback. Only used when buy/sell odds are not provided. New integrations should use `buy_decimal_odds` and `sell_decimal_odds`. |
| `status` | Yes | enum | `active` or `suspended`. |

`buy_decimal_odds` and `sell_decimal_odds` are two independent odds values:

- `buy_decimal_odds`: price for a user buying this outcome.
- `sell_decimal_odds`: price for a user selling or closing this outcome.
- TurboFlow separately matches best multi-vendor prices for the buy side and sell side internally.

## 5. WebSocket Odds Push

SIG continuously pushes odds frames to TurboFlow over WebSocket.

The WebSocket stream is the only real-time channel. It is not an event bus and does not require an event-type wrapper.

### 5.1 Connection

```text
# UAT: connect to TurboFlow WebSocket server. Authentication: see section 7.1.
wss://api.turboflow-test.xyz/api/v1/vendor/feed

# Recommended header-based credentials:
API-KEY / SIGN / TIMESTAMP

# Fallback query-string credentials when a WebSocket client cannot set headers:
wss://api.turboflow-test.xyz/api/v1/vendor/feed?api_key=<pubkey>&sign=<sig>&timestamp=<unix_sec>

# Current UAT credential example:
API_KEY=7f457c3bb3940ed42f665a1135ed2a21cc41191d2dd4e66aada00fb40030edf4
PRIVATE_SEED=05285997edac7c463232bf1bb02ba8a26589bb4ce9e1d86b998a88a20d97caad
```

Protocol:

- `wss` over TLS 1.2 or above.
- Authentication uses the Ed25519 signature described in section 7.1.
- Headers: `API-KEY / SIGN / TIMESTAMP`.
- Query-string fallback is only for WebSocket clients that cannot set headers.
- The old `?token=` authentication method is no longer used.

Heartbeat:

- Both sides send `ping` / `pong` every 30 seconds.
- If no upstream frame is received within 60 seconds, the connection is disconnected and reconnected.

Reconnect:

- SIG should reconnect with exponential backoff after disconnection.
- TurboFlow treats reconnects as normal behavior.

After SIG connects successfully, TurboFlow immediately sends the first frame with the list of currently available markets. The `market_info` schema follows the Polymarket-style hierarchy: one match is one event; markets are grouped by `market_code`; each tradable binary outcome is represented as one market with a stable slug.

### 5.4 Push Frequency

Rules:

- Push whenever odds change.
- Each market must send at least one frame every 5 seconds.
- Each frame is a full market snapshot, not a delta update.
- SIG does not need to replay historical frames.
- TurboFlow discards old frames by `at_ms`.

### 5.5 Heartbeat

```json
{ "op": "ping", "at": "2026-05-19T10:00:00Z" }
```

```json
{ "op": "pong", "at": "2026-05-19T10:00:00Z" }
```

Either side may disconnect and reconnect if no peer frame is received within 60 seconds.

## 7. Authentication and Rate Limits

### 7.1 Credentials

Authentication is unified with the TurboFlow main-site API signing scheme. See `turboflow-api-doc` signature generation.

The vendor feed uses Ed25519 asymmetric signatures:

- Vendor holds the private key.
- TurboFlow stores the public key.
- The `vendor_api_keys.api_key` column stores the 64-character hex public key.

HTTP header fields:

| Header | Description |
| --- | --- |
| `API-KEY` | Vendor Ed25519 public key, 64-character hex. |
| `SIGN` | Request signature, hex string. |
| `TIMESTAMP` | Unix seconds timestamp. Tolerance: +/- 300 seconds. |

When the WebSocket client cannot set headers, the same fields may be passed through query parameters:

```text
?api_key=<pubkey>&sign=<signature_hex>&timestamp=<unix_sec>
```

Signing string:

```text
method=GET&path={path}&timestamp={timestamp}&access-key={apiKey}
```

Rules:

- `method` is always `GET` for vendor feed endpoints.
- `path` is the pure URL path and must not include query parameters.
- `timestamp` must be identical to the `TIMESTAMP` header or query parameter.
- `access-key` must be identical to `API-KEY` / `api_key`.

Signature generation steps:

1. Construct the signing string.
2. Use the public key after hex decoding to calculate the HMAC-SHA256 digest of the signing string.
3. Sign the digest with the vendor Ed25519 private key.
4. Put the signature hex into `SIGN`.

Operational conventions:

- Each environment has an isolated keypair: dev / sit / uat / prod.
- Operations writes the vendor public key into `vendor_api_keys`.
- Setting `enabled=false` invalidates the key within the maximum 60-second reload window.
- The private key is held by the vendor and must not be committed to Git, YAML, or configuration files.
- `scripts/genvendorkey` can be used to generate a keypair. Public key goes to the database; private key is delivered to the vendor.

### 7.2 Rate Limits

| Interface | Limit |
| --- | --- |
| `GET /v1/odds/snapshot` | 10 QPS |
| `GET /v1/odds/match` | 30 QPS |
| WebSocket connections | Single API Key <= 4 connections |

When the REST rate limit is exceeded:

- Return HTTP 429.
- Include `Retry-After`.

When the WebSocket connection limit is exceeded:

- Reject the connection directly.

## 11. Changelog

### v2.3.1 - 2026-05-29

- `market_info` now uses a Polymarket-style slug schema: one match is one event, `market_groups[]` groups binary markets by `market_code`, and each binary market carries a stable `slug` for `result_1x2`, `result_ht`, and future market groups.

### v2.3 - 2026-05-29

- Authentication is unified with the TurboFlow main-site Ed25519 signature scheme. The vendor feed now uses `API-KEY / SIGN / TIMESTAMP`, replacing the previous Bearer token / `?token=` approach.
- Outcome odds are split into `buy_decimal_odds` and `sell_decimal_odds`. The legacy `decimal_odds` field is retained as a fallback only.
- After a vendor connects to `/api/v1/vendor/feed`, TurboFlow sends `market_info` as the first frame to provide the available market snapshot.
- `GET /api/v1/vendor/markets` provides the same market list query capability.

### v2.2 - 2026-05-19

- Rewritten as an odds-data-only integration. SIG only provides odds data and no longer needs to handle RFQ, trade, acceptance, or settlement.
- Removed `quote`, `trade`, and `accept` flows.
- Removed `provider_quote_id`, `trade_id`, and `market_id`.
- WebSocket frames directly carry odds snapshots without an event wrapper.
- `league_code`, `market_code`, and `outcome_key` are fixed by TurboFlow.
