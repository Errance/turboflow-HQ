## 2. Identifier System

| Field | Defined By | Type | Description |
| --- | --- | --- | --- |
| `league_code` | TurboFlow, section 2 | string <= 32 | Competition code from the fixed table. |
| `sig_match_id` | SIG | string <= 64 | SIG internal unique match ID. Stable for each match. |
| `market_code` | TurboFlow, section 4 | string <= 32 | Market type code, for example `result_1x2`. |
| `outcome_key` | TurboFlow, section 4 | string <= 32 | Stable outcome key, for example `home_win`. |
| `kickoff_at` | SIG | RFC3339 UTC | Match kickoff time. |
| `at` / `at_ms` | SIG | RFC3339 / int64 ms | Data production time, used to discard out-of-order frames. |

Notes:

- The following fields are no longer used: `provider_market_id`, `provider_quote_id`, `provider_trade_id`, `provider_selection_id`, and similar provider-side identifiers. TurboFlow business logic uniquely identifies entries by `(league_code, sig_match_id, market_code, outcome_key)`.
- The same match must keep the same `sig_match_id` across all SIG pushes. Changing it is treated as an error.
- All time values must use UTC.

## 4. Market Code and Outcome Key

SIG must not customize `market_code` or `outcome_key`. SIG must use the values in the tables below. Markets not listed here do not need to be supported by SIG.

### 4.1 Match-Level Markets (`scope = match`)

| market_code | Market | outcome_key List | Description |
| --- | --- | --- | --- |
| `result_1x2` | Full-time result | `home_win` / `draw` / `away_win` | - |
| `result_ht` | Half-time result | `home_win_ht` / `draw_ht` / `away_win_ht` | - |
| `btts` | Both teams to score | `yes` / `no` | both-teams-to-score |
| `over_under_2_5` | Over / under 2.5 goals | `over` / `under` | Total goals versus 2.5 |
| `over_under_1_5` | Over / under 1.5 goals | `over` / `under` | - |
| `over_under_3_5` | Over / under 3.5 goals | `over` / `under` | - |
| `correct_score` | Correct score | `s_<H>_<A>`, for example `s_2_1` | Home goals - away goals |
| `next_goal` | Next goal | `home_win` / `away_win` / `no_goal` | - |

### 4.2 Series-Level Markets (`scope = series`)

Binary YES/NO model. Each candidate is an independent market.

| market_code | Market | outcome_key List | `candidate_id` Meaning |
| --- | --- | --- | --- |
| `series_champion` | Champion | `yes` / `no` | Team / country code. SIG uses its own `team_id`; TurboFlow maintains the mapping. |
| `series_top_scorer` | Top scorer | `yes` / `no` | Player ID |
| `series_top_4` | Top 4 | `yes` / `no` | Team ID |
| `series_relegation` | Relegation | `yes` / `no` | Team ID |
| `series_group_first` | Group winner | `yes` / `no` | Team ID |
| `series_group_qualify` | Group qualification | `yes` / `no` | Team ID |

### 4.3 Outcome Common Fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `outcome_key` | string | Yes | Must follow section 4.1 / 4.2. |
| `buy_decimal_odds` | string | Yes | Buy odds, decimal format with 6 decimal places, for example `"2.150000"`. |
| `sell_decimal_odds` | string | Yes | Sell odds, decimal format with 6 decimal places, for example `"1.950000"`. |
| `decimal_odds` | string | No | Legacy single-odds compatibility field. Only used as fallback when buy/sell odds are missing. New integrations should use buy/sell odds. |
| `status` | enum | Yes | `active` / `suspended`. `suspended` means betting is paused, while odds may still be pushed for display. |

Buy and sell are two independent odds: buy odds are used when a user buys this outcome, and sell odds are used when a user sells or closes this outcome. TurboFlow internally matches the best multi-vendor price separately on the buy side and sell side.

## 5. WebSocket Real-Time Odds Stream

SIG continuously pushes odds data frames through WebSocket. This is the only real-time channel. It is not an event bus and has no event-type wrapper.

### 5.1 Connection

```text
# UAT: connect to the TurboFlow WebSocket server. Authentication: see section 7.1.
wss://api.turboflow-test.xyz/api/v1/vendor/feed

# Recommended credential transport through headers:
API-KEY / SIGN / TIMESTAMP

# Query-string fallback when the WebSocket client cannot set headers:
wss://api.turboflow-test.xyz/api/v1/vendor/feed?api_key=<pubkey>&sign=<sig>&timestamp=<unix_sec>
```

- Protocol: WebSocket over `wss://`, TLS 1.2 or above.
- Authentication: Ed25519 signature. See section 7.1, `API-KEY / SIGN / TIMESTAMP`. The old `?token=` authentication method is no longer used.
- Heartbeat: both sides send `ping` / `pong` every 30 seconds. If there is no upstream frame for 60 seconds, disconnect.
- Reconnect: the connecting side should use exponential backoff: 1s / 2s / 4s / 8s, capped at 30s.

Connection first frame: `market_info` snapshot.

After the connection is established, TurboFlow immediately sends one frame containing the current available market snapshot. Each `markets[]` element corresponds to one Polymarket orderbook with a `slug`, not an event-level aggregation. SIG prices and pushes odds by `slug`.

```json
{
  "op": "market_info",
  "at_ms": 1748764830123,
  "events": [
    {
      "league_code": "FIF",
      "match_id": "1001",
      "event_slug": "fif-egy-rus-2026-05-28",
      "polymarket_event_slug": "fif-egy-rus-2026-05-28",
      "sig_match_id": "sig:fif:2026-05-28:egy-rus",
      "status": "incoming",
      "home_team_slug": "egy",
      "away_team_slug": "rus",
      "kickoff_at": "2026-05-28T19:30:00Z",
      "market_groups": [
        {
          "market_code": "result_1x2",
          "market_name": "Full Time Result",
          "markets": [
            {
              "outcome_key": "home_win",
              "outcome_label": "Egypt",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-egy"
            },
            {
              "outcome_key": "draw",
              "outcome_label": "Draw",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-draw"
            },
            {
              "outcome_key": "away_win",
              "outcome_label": "Russia",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-rus"
            }
          ]
        },
        {
          "market_code": "result_ht",
          "market_name": "Half Time Result",
          "markets": [
            {
              "outcome_key": "home_win_ht",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-halftime-result-home"
            },
            {
              "outcome_key": "draw_ht",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-halftime-result-draw"
            },
            {
              "outcome_key": "away_win_ht",
              "side": "yes",
              "slug": "fif-egy-rus-2026-05-28-halftime-result-away"
            }
          ]
        }
      ]
    }
  ]
}
```

Slug rules, consistent with Polymarket. `event_slug` comes from the confirmed `polymarket_event_link`.

| Level | Rule |
| --- | --- |
| `event_slug` | `<league>-<home>-<away>-<yyyy-mm-dd>` |
| `status` | `incoming` = not started; `open` = in progress, including live / halftime. |
| `result_1x2` | `home_win` -> `<event_slug>-<home>`; `draw` -> `<event_slug>-draw`; `away_win` -> `<event_slug>-<away>`. |
| `result_ht` | `home_win_ht` -> `<event_slug>-halftime-result-home`; `draw_ht` -> `...-draw`; `away_win_ht` -> `...-away`. |

For matches already linked to a Polymarket event, TurboFlow preferentially uses the confirmed slug. For unlinked matches, TurboFlow falls back to generating `event_slug` from internal match metadata using the same rule above; `league_code` is taken from `match_league.short_name`.

`event_slug` and `polymarket_event_slug`, for SIG precise matching:

| Field | Required | Description |
| --- | --- | --- |
| `event_slug` | Yes | Internal routing slug. If the match is linked to Polymarket, this is the confirmed slug. If not linked, it is generated from match metadata, and the `<league>` prefix is normalized to the Polymarket prefix using known mappings, for example `wc -> fifwc` and `intl -> fif`. |
| `polymarket_event_slug` | Yes, value may be empty | Filled with the original Polymarket slug only when TurboFlow has confirmed that this market is bound to Polymarket. Used by SIG for exact matching. Empty means TurboFlow has not confirmed that this market exists on Polymarket. |

SIG matching recommendation: when `polymarket_event_slug` is non-empty, match by it exactly because it is authoritative. When it is empty, fall back to `event_slug`; its prefix is already aligned to Polymarket as much as possible. Confirmed internal `short_name` to Polymarket prefix mappings are currently: `WC -> fifwc`, `INTL -> fif`. Leagues whose prefixes already match do not need registration.

The same structure can also be queried at any time through `GET /api/v1/vendor/markets`. The response is:

```json
{
  "events": []
}
```

### 5.2 Subscription

After connection, TurboFlow sends one subscription request. SIG then continuously pushes odds.

```json
{
  "op": "sub",
  "league_codes": ["EPL", "UCL", "WC"]
}
```

| Field | Required | Description |
| --- | --- | --- |
| `op` | Yes | Fixed value: `sub`. Future extensions may include `unsub` / `replace`. |
| `league_codes` | Yes | Subscribed competition code list. An empty array means subscribe to all. |

SIG response:

```json
{
  "op": "sub_ack",
  "league_codes": ["EPL", "UCL", "WC"],
  "at": "2026-05-19T10:00:00Z"
}
```

### 5.3 Odds Data Frame

SIG actively pushes odds frames. Single frame = one odds snapshot for one outcome. Since v3.4, each frame returns only one `outcome` object and no longer uses the `outcomes` array. There is no `type` / `event_id` wrapper. TurboFlow deduplicates and sorts by `(league_code, sig_match_id, market_code, outcome_key, at_ms)`.

```json
{
  "league_code": "EPL",
  "sig_match_id": "sig-20260601-arsenal-chelsea",
  "polymarket_event_id": "0x9a3f...e21",
  "scope": "match",
  "kickoff_at": "2026-06-01T19:00:00Z",
  "market_code": "result_1x2",
  "market_status": "open",
  "outcome": {
    "outcome_key": "home_win",
    "buy_decimal_odds": "2.150000",
    "sell_decimal_odds": "1.950000",
    "status": "active"
  },
  "at_ms": 1748764830123
}
```

Multiple outcomes under the same market, such as `home_win` / `draw` / `away_win`, are split into multiple frames. Each frame contains one outcome.

Compatibility: old frames with the `outcomes` array are still accepted, and REST fallback snapshots may still return a whole market with an array. WebSocket real-time pushes should use the single-outcome format.

#### Field Description

| Field | Required | Description |
| --- | --- | --- |
| `league_code` | Yes | Must come from the table in section 2. |
| `sig_match_id` | Required when `scope=match` | Recommended for `scope=series` when an associated match exists; may be empty. |
| `polymarket_event_id` | No | SIG-side Polymarket event ID. Only passed through for reconciliation and does not participate in identification. Identification still uses the four-tuple. Include it when SIG has it; otherwise omit it. |
| `scope` | Yes | `match` / `series`. |
| `kickoff_at` | Required when `scope=match` | Match kickoff time. |
| `market_code` | Yes | Must come from the table in section 4. |
| `market_status` | Yes | `open` / `closed`. `closed` means the market no longer accepts betting. |
| `outcome` | Yes | Latest odds snapshot for a single outcome. Fields follow section 4.3. |
| `at_ms` | Yes | Timestamp when SIG produced this snapshot, in UTC milliseconds. |

#### `scope=series` Example

```json
{
  "league_code": "WC",
  "scope": "series",
  "market_code": "series_champion",
  "candidate_id": "team_argentina",
  "candidate_label": "Argentina",
  "market_status": "open",
  "outcome": {
    "outcome_key": "yes",
    "buy_decimal_odds": "5.200000",
    "sell_decimal_odds": "4.900000",
    "status": "active"
  },
  "at_ms": 1748764830123
}
```

| Extra Field | Description |
| --- | --- |
| `candidate_id` | Required when `scope=series`; for example team / player ID. |
| `candidate_label` | Required when `scope=series`; display text. |

#### Closed Market Example

When closing a market, set `market_status=closed`. TurboFlow will set all outcomes under this market to `suspended` at once. SIG does not need to push each outcome individually. `outcome` may still carry the final odds for display only.

```json
{
  "league_code": "EPL",
  "sig_match_id": "sig-20260601-arsenal-chelsea",
  "scope": "match",
  "market_code": "result_1x2",
  "market_status": "closed",
  "outcome": {
    "outcome_key": "home_win",
    "buy_decimal_odds": "1.910000",
    "sell_decimal_odds": "1.750000",
    "status": "suspended"
  },
  "at_ms": 1748767200000
}
```

### 5.4 Push Frequency

- Push whenever odds change.
- SIG should also guarantee at least one frame per outcome every 5 seconds, even when there is no change. This acts as the active heartbeat for that outcome.
- Each frame contains only one outcome. Multiple outcomes under the same market are split into multiple frames.
- Historical replay / resend is not allowed. Do not backfill old frames. TurboFlow strictly discards old frames by `at_ms`.

### 5.5 Heartbeat

```json
// SIG -> TurboFlow
{
  "op": "ping",
  "at_ms": 1748767200000
}
```

```json
// TurboFlow -> SIG
{
  "op": "pong",
  "at_ms": 1748767200005
}
```

Either side disconnects and reconnects if no peer frame is received within 60 seconds.

## 6. REST Startup Sync / Disconnect Completion

REST is only a fallback channel. It is not a real-time channel. During normal operation, only WebSocket is used.

### 6.1 Full Odds by Competition

```http
GET /v1/odds/snapshot?league_code=EPL
```

Response: same logical format as WebSocket data frames, wrapped by an outer array.

```json
{
  "league_code": "EPL",
  "items": [
    {
      "sig_match_id": "...",
      "scope": "match",
      "market_code": "result_1x2",
      "market_status": "open",
      "outcomes": [],
      "at_ms": 1748764830123
    },
    {
      "sig_match_id": "...",
      "scope": "match",
      "market_code": "btts",
      "market_status": "open",
      "outcomes": [],
      "at_ms": 1748764830200
    }
  ]

```

### 6.2 All Markets for a Single Match

```http
GET /v1/odds/match?league_code=EPL&sig_match_id=sig-20260601-arsenal-chelsea
```

The response follows the same format as above. `items[]` contains all markets for that match only.

## 7. Authentication and Rate Limits

### 7.1 Credentials: Ed25519 Signature, Unified with Main Site

Authentication is unified with the TurboFlow main-site API signature generation described in `turboflow-api-doc signature-generation`. The authentication method is Ed25519 asymmetric signature. The vendor holds the private key, and TurboFlow stores the public key in the `vendor_api_keys.api_key` column as a 64-character hex public key.

Request headers. If the WebSocket client cannot set headers, fallback query parameters with the same names may be used: `api_key` / `sign` / `timestamp`.

| Header | Description |
| --- | --- |
| `API-KEY` | Vendor Ed25519 public key, 64-character hex. |
| `SIGN` | Request signature, hex. |
| `TIMESTAMP` | Unix timestamp in seconds. Tolerance: +/- 300 seconds to prevent replay. |

Signing string, same as the main site. Vendor feed endpoints are all `GET` and have no body. `path` is the pure URL path and excludes query parameters.

```text
method=GET&path={path}&timestamp={timestamp}&access-key={apiKey}
```

Signature steps:

1. Construct the signing string above.
2. Use the public key after hex decoding to calculate HMAC-SHA256 over the signing string and obtain the digest.
3. Use the vendor Ed25519 private key to sign the digest.
4. Convert the signature result to hex and put it into `SIGN`.

Conventions:

- Keys are isolated by environment: dev / sit / uat / prod each has a separate keypair.
- Operations writes the public key into `vendor_api_keys`. Setting `enabled=false` invalidates it immediately, with a maximum 60-second reload window.
- The vendor keeps the private key. It must not be committed into Git, YAML, or any configuration file.
- `scripts/genvendorkey` can be used to generate a keypair. The public key goes into the database, and the private key is delivered to the vendor.

### 7.2 Rate Limits

| Interface | Default |
| --- | --- |
| `GET /v1/odds/snapshot` | 10 QPS. Only for startup / completion. |
| `GET /v1/odds/match` | 30 QPS |
| WebSocket connections | Single API Key <= 4 connections |

When the REST limit is exceeded, return HTTP 429 with `Retry-After`. When the WebSocket connection count is exceeded, reject the connection directly.
