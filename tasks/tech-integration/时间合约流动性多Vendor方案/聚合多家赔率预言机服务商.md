# 聚合多家赔率预言机服务商

> 面向对象：TurboFlow 内部工程团队
> 目标：将当前单一 SIG/SCB 流动性报价模型升级为多 Vendor 报价推送、多 Vendor 专属流动性池路由模型。
> 当前结论：新 Vendor 以接入 TurboFlow inbound WebSocket quote API 为主；现有 SIG/SCB outbound WebSocket adapter 仅保留作平滑迁移，不作为未来 Vendor 扩展模式。

## 1. 背景与目标

当前猜涨跌产品中，SIG/SCB 是唯一外部流动性报价 Vendor。TurboFlow 通过 outbound WebSocket 连接 SIG/SCB feed，订阅 `BTC-30s`、`BTC-1m`、`BTC-3m`、`BTC-5m`、`BTC-15m`、`BTC-1h`、`ETH-30s`、`ETH-1m`、`ETH-3m`、`ETH-5m`、`ETH-15m`、`ETH-1h` 等 up/down market odds。

SIG/SCB 当前 WebSocket 协议：

- endpoint: `/ws/public`
- handshake headers:
  - `X-API-KEY`
  - `X-API-TS`
  - `X-API-SIGN`
- signature:

```text
signature = HMAC_SHA256(secret, apiKey + timestamp)
```

订阅示例：

```json
{
  "t_market": ["BTC-5m", "ETH-3m"],
  "type": "market"
}
```

报价示例：

```json
{
  "event_type": "market-up-down",
  "t_market": "BTC-5m",
  "ts": "1757908892351",
  "t_outcomes": [
    {
      "t_outcome": "BTC-5m-up",
      "return_rate": 0.8
    },
    {
      "t_outcome": "BTC-5m-down",
      "return_rate": 0.8
    }
  ]
}
```

本次升级目标：

1. TurboFlow 提供 inbound WebSocket quote API，让多 Vendor 主动推送猜涨跌 odds。
2. 新 Vendor 不再需要 TurboFlow 为每家 Vendor 写 outbound adapter。
3. 内部将 Vendor 报价标准化，建立多 Vendor quote book。
4. 下单时根据可配置 routing strategy 选择最优报价，并立即路由到对应 Vendor 的专属 liquidity pool/account。
5. 保留 legacy SIG/SCB adapter 并接入同一标准化 quote ingestion path，迁移完成后可下线。
6. Auth method 与 quote win reaction speed 作为开放讨论项，不在本方案中锁死。

## 2. 当前代码链路调查

相关 repo：

- `surfv2-dex-svm-keeper`
- `cex-api-service`
- `base`

  - `handleSigWsMsg` 解析 SIG/SCB odds，将 up/down `return_rate` 写入缓存。
  - `procPredictConfigs` 发布猜涨跌市场配置。
- `surfv2-dex-svm-keeper/common/websocket.go`
  - `SubscribeWs`
  - `WsSigMarketMsg`
  - `PredictRRateRedisVo`
- `surfv2-dex-svm-keeper/domain/repo/repo_redis.go`
  - `SavePredictDurationReturnRate`
  - `GetPredictDurationReturnRate`
- `surfv2-dex-svm-keeper/domain/services/api_order_service.go`
  - `CreateOrderForPredictMarket`
  - 当前启用外部报价时订单池子设置为 `common.PredictOtherPoolID`
- `surfv2-dex-svm-keeper/common/constant.go`
  - `PredictMarketPoolID = 101`
  - `PredictOtherPoolID = 102`
- `base/rpc/predict_market.go`
  - `PredictPublishMsg`
  - `PredictOrderRisk`

当前限制：

- Vendor identity 没有进入报价缓存与下单决策。
- 外部流动性池只有固定 `PredictOtherPoolID = 102` 语义。
- 报价选择是单 Vendor overwrite，不支持多个 Vendor 同时报价、比较和审计。
- 订单只持久化 `ReturnRate` 和 `PoolID`，缺少 `vendor_id`、`quote_id`、routing strategy version、decision reason 等审计字段。
- 新 Vendor 如果沿用当前模式，需要为每家 Vendor 写 outbound adapter，不利于扩展。

## 3. 推荐方案

采用方案：Quote Gateway + normalized quote book + pluggable router。

目标架构：

```text
Vendor
  -> TurboFlow inbound WebSocket quote API
  -> auth/session validation
  -> quote normalization
  -> latest quote book by vendor + market + duration + side
  -> best-quote routing strategy
  -> selected vendor liquidity pool/account
  -> existing predict order open/settle flow
```

Legacy SIG/SCB 迁移链路：

```text
SIG/SCB outbound WS adapter
  -> same normalized quote ingestion interface
  -> quote book
  -> router
```

原则：

- 新 Vendor 统一接入 TurboFlow inbound WebSocket quote API。
- 不为新 Vendor 增加 outbound subscription adapter。
- SIG/SCB 当前 outbound adapter 保留为 `sig_legacy` source，用于平滑迁移。
- 下单时选择 winning quote 后，立即使用该 Vendor 对应的专属 liquidity pool/account。
- Routing strategy 必须是可扩展的，默认实现为 best return rate。

## 4. 组件设计

### 4.1 Vendor Registry

维护 Vendor、市场权限、池子映射和风控配置。

建议字段：

```text
vendor_id
vendor_name
status
allowed_pairs
allowed_durations
pool_id
max_quote_age_ms
clock_skew_ms
priority
fallback_enabled
auth_config_ref
created_at
updated_at
```

Phase 1 可使用 `sys_config` JSON 快速落地；如果团队需要更好的审计、后台管理和热更新，建议建 DB table。

每个可参与 routing 的 Vendor 必须先完成专属 liquidity pool/account 配置。Router 不应允许没有 `pool_id`、pool disabled、pool 不健康或 pool capacity 不足的 Vendor 赢得报价。

示例配置：

```json
{
  "vendors": [
    {
      "vendor_id": "scb",
      "vendor_name": "SCB",
      "status": "enabled",
      "pool_id": 102,
      "allowed_markets": ["BTC-30s", "BTC-1m", "BTC-3m", "BTC-5m", "BTC-15m", "BTC-1h", "ETH-30s", "ETH-1m", "ETH-3m", "ETH-5m", "ETH-15m", "ETH-1h"],
      "max_quote_age_ms": 2000,
      "clock_skew_ms": 5000,
      "priority": 10
    }
  ]
}
```

### 4.2 Inbound Quote WebSocket Gateway

新增 Vendor-facing WebSocket endpoint：

```text
/ws/vendor/predict-quotes
```

建议 Phase 1 放在 `surfv2-dex-svm-keeper`，原因：

- keeper 已经持有猜涨跌 order/risk/pool routing 逻辑。
- 可以直接复用当前 `PredictMarketService`、Redis、routing 与 order creation 链路。
- 改造范围比新增独立服务更小。

后续如果 Vendor 数量、连接规模、权限管理复杂度提高，可以将该 gateway 抽成独立服务。

### 4.3 Quote Normalizer

将所有来源的 Vendor quote 转成内部统一结构：

```text
vendor_id
quote_source
vendor_quote_id
market
pair_id
duration
side
return_rate
quote_ts
received_at
pool_id
raw_payload_hash
```

其中：

- `quote_source`: `inbound_ws` / `sig_legacy`
- `side`: `up` / `down`，内部映射到当前 `OpenLong` / `OpenShort` 使用的 bid/ask return rate。
- `return_rate`: 当前先沿用现有范围校验，`0 < return_rate <= 1`。

### 4.4 Quote Book

保存每个 Vendor 对每个 `pair + duration + side` 的最新有效报价。

建议 Redis key：

```text
pm:vendor_quotes:{pair_id}:{duration}:{side}
```

Hash field:

```text
{vendor_id}
```

Hash value:

```json
{
  "vendor_id": "scb",
  "vendor_quote_id": "scb-1757908892351-001",
  "market": "BTC-5m",
  "pair_id": 1,
  "duration": 300,
  "side": "up",
  "return_rate": "0.80",
  "quote_ts": 1757908892351,
  "received_at": 1757908892399,
  "pool_id": 102,
  "source": "inbound_ws"
}
```

TTL 建议：

- Redis key TTL 应大于 `max_quote_age_ms`，例如 10-30 秒。
- Router 仍必须基于 quote age 判断有效性，不能只依赖 Redis TTL。

### 4.5 Routing Strategy

新增 routing layer：

```text
SelectQuote(pair_id, duration, order_way, amount, now) -> RouteDecision
```

返回：

```text
vendor_id
pool_id
return_rate
quote_id
quote_ts
quote_received_at
routing_strategy
decision_reason
```

默认策略：

1. 按 `pair_id + duration + side` 找候选 quotes。
2. 过滤：
   - Vendor enabled
   - market/duration enabled
   - quote fresh
   - pool enabled/healthy
   - pool/vendor capacity 足够
3. 选择最高 `return_rate`。
4. tie-breaker：
   - Vendor priority 更高者优先。
   - `received_at` 更新者优先。

可扩展策略：

- best quote
- sticky winner
- weighted vendor
- capacity-aware
- risk-adjusted quote
- manual priority/fallback

### 4.6 Public Predict Config Publishing

当前前端/API 侧通过 `PredictPublishMsg` 消费猜涨跌配置，其中每个 duration 暴露单一 `AskReturnRate` / `BidReturnRate`。多 Vendor 后，公开配置仍建议保持这个形状：对每个 `pair + duration + side` 发布当前 router 选出的 winning quote，而不是把所有 Vendor quote 暴露给客户端。

原因：

- 客户端只需要展示当前可成交赔率。
- 最终下单仍由服务端重新执行 routing 并锁定 quote，不能信任客户端提交的 `return_rate`。
- 后续如果业务需要展示 Vendor depth 或 best/second-best spread，可以新增内部管理接口或扩展 public payload。

发布逻辑：

```text
quote book update
  -> recompute current winner for affected market/side
  -> update globalRiskCfg / PredictPublishMsg
  -> publish existing predict_risk_configs event
```

如果 Phase 1 为了降低改造量，也可以继续由 `procPredictConfigs` 定时发布，但下单路径必须实时调用 router，避免只依赖公开配置缓存。

## 5. Vendor WebSocket API

Endpoint：

```text
/ws/vendor/predict-quotes
```

### 5.1 Connection

Vendor 建立 WebSocket 连接后：

1. TurboFlow 校验 handshake auth。
2. Vendor 可发送 `hello` / `register` frame 声明支持 markets；如果 registry 已配置完整权限，该步骤可选。
3. Vendor 持续发送 `quote_batch`。
4. TurboFlow 对 accepted/rejected quote 返回 ack。
5. Vendor 定期 ping/pong 或发送 heartbeat。

### 5.2 Quote push payload

建议 payload：

```json
{
  "type": "quote_batch",
  "vendor_quote_id": "scb-1757908892351-001",
  "sent_at": 1757908892351,
  "quotes": [
    {
      "market": "BTC-5m",
      "outcomes": [
        {
          "outcome": "up",
          "return_rate": "0.80"
        },
        {
          "outcome": "down",
          "return_rate": "0.79"
        }
      ]
    }
  ]
}
```

Ack：

```json
{
  "type": "ack",
  "vendor_quote_id": "scb-1757908892351-001",
  "accepted": 2,
  "rejected": 0,
  "server_ts": 1757908892399
}
```

Partial reject：

```json
{
  "type": "ack",
  "vendor_quote_id": "scb-1757908892351-001",
  "accepted": 1,
  "rejected": 1,
  "server_ts": 1757908892399,
  "errors": [
    {
      "market": "BTC-5m",
      "outcome": "down",
      "code": "QUOTE_OUT_OF_RANGE",
      "message": "return_rate must be > 0 and <= 1"
    }
  ]
}
```

### 5.3 Validation

每条 quote 需要校验：

- `vendor_quote_id` 非空且在 `vendor_id + vendor_quote_id` 下幂等。
- `sent_at` 在允许 clock skew 内。
- `market` 可解析为 supported `pair_id + duration`。
- Vendor 对该 market/duration 有权限。
- `return_rate > 0 && return_rate <= 1`。
- quote 中至少包含一个 outcome；推荐每次同时包含 up/down。

连接级违规：

- Auth 失败。
- Vendor disabled。
- 消息过大或无法解析。
- 重复异常或 reject ratio 过高。

## 6. 下单集成

当前 `CreateOrderForPredictMarket` 根据 `pairOrderConfig.BidReturnRate / AskReturnRate` 设置订单赔率，并在 `keeperPredictEnableSubscribe = true` 时将订单池子切到 `PredictOtherPoolID = 102`。

改造后：

1. 订单校验阶段仍使用全局 predict market config 验证 pair/duration/min/max。
2. 创建订单前调用 routing layer：

```text
route = SelectQuote(pair_id, duration, order_way, usdt_value, submit_at)
```

3. 使用 `route.return_rate` 设置 `order.ReturnRate`。
4. 使用 `route.pool_id` 设置 `order.PoolID`。
5. 持久化 route metadata。
6. 后续 open/settle 链上流程继续使用订单上的 `PoolID` 与 `ReturnRate`，保证 settlement 使用开仓时锁定的报价与池子。

建议新增订单字段或订单 route audit 表：

```text
liquidity_vendor_id
liquidity_pool_id
quote_id
quote_ts
quote_received_at
quote_source
routing_strategy
routing_decision_reason
```

如果不希望立刻改大 `chain_predict_order` 表，可以新增 `chain_predict_order_liquidity_route` 表：

```text
id
order_id
vendor_id
pool_id
quote_id
quote_ts
quote_received_at
quote_source
return_rate
routing_strategy
decision_reason
raw_quote_snapshot
created_at
```

建议优先使用 route audit 表，减少对主订单表的侵入，同时满足审计要求。

## 7. Routing 速度开放讨论

本方案不锁定 quote win reaction speed，需要产品、风控、技术共同确认。

### Option A: Immediate switching

每次下单都读取当前最新 quote book，最高 `return_rate` Vendor 获胜。

优点：

- 用户理论上拿到最优赔率。
- 实现简单，UAT 验证直观。

风险：

- 高频报价下 pool route 可能快速切换。
- Vendor exposure 与池子 PnL 变化更快，需要监控支持。

### Option B: Sticky winner window

某个 Vendor 赢得 `market + side` 后，在 N 秒窗口内保持 winner，除非 quote stale、Vendor 断连或 quote 明显变差。

优点：

- 降低 route oscillation。
- 对池子会计和 Vendor exposure 更稳定。

风险：

- 部分订单可能没有拿到瞬时最优 quote。

### Option C: Threshold / hysteresis

只有当新 Vendor 比当前 winner 高出配置阈值，才切换 winner。

优点：

- 兼顾价格优化与稳定性。
- 适合生产长期运行。

风险：

- 需要确定阈值单位和监控。

建议：

- UAT 先使用 immediate switching，加完整 route decision logs。
- 生产前根据 Vendor 报价频率、route oscillation 指标和运营偏好，决定是否启用 sticky/hysteresis。

## 8. Auth method 开放讨论

Inbound vendor WebSocket auth 需要与 infra/security 最终确认。

### Option A: HMAC headers

沿用当前 SIG/SCB 风格：

```text
X-API-KEY
X-API-TS
X-API-SIGN = HMAC_SHA256(secret, apiKey + timestamp)
```

优点：

- 当前代码已有 `GenSignature` 类似 helper。
- Vendor 容易理解，和现有 SIG/SCB 文档一致。
- Phase 1 落地最快。

注意：

- 需要服务端校验 clock skew。
- secret 存 KMS/Secrets Manager。
- 建议支持 key rotation。

### Option B: PrivateLink / allowlist + API key

通过网络边界降低暴露面，再用 API key 识别 Vendor。

优点：

- 对私有网络集成友好。

风险：

- 单独 API key 不足以防 replay；仍建议配合签名。

### Option C: mTLS

Vendor 使用 client certificate 建连。

优点：

- 安全边界强。

风险：

- 证书生命周期和 Vendor 对接成本更高。

### Option D: JWT / client credentials

Vendor 先换 token，再用 token 建 WS。

优点：

- 权限模型清晰，可扩展。

风险：

- Phase 1 增加额外服务与 token 生命周期复杂度。

建议：

- Phase 1 使用 HMAC headers + optional network allowlist。
- 后续如 Vendor 数量和安全要求提高，再升级到 mTLS 或 client credentials。

## 9. Legacy SIG/SCB 迁移策略

迁移期间：

- 当前 outbound SIG/SCB subscriber 继续运行。
- 将 `handleSigWsMsg` 改造为调用统一 quote normalizer。
- SIG/SCB source 使用固定 `vendor_id = sig_legacy` 或 `scb_legacy`。
- SIG/SCB 对应 pool 通过 Vendor Registry 配置。
- Router 不关心 quote 来源，只按标准 quote book 选择。

迁移完成条件：

- 新 inbound WS Vendor 在 UAT 和 Production 稳定推送。
- 当前 SIG/SCB Vendor 已改为 inbound push，或业务确认不再依赖 legacy feed。
- 监控显示 quote freshness、order route、pool settlement 无异常。
- 删除或关闭 legacy adapter config。

## 10. Observability 与告警

Vendor 维度：

- active sessions
- last quote received time
- accepted quote count
- rejected quote count
- reject reasons
- heartbeat timeout
- reconnect count

Market 维度：

- current winning vendor
- winning return rate
- second-best return rate
- quote age
- route switch count
- no valid quote count

Order 维度：

- selected vendor
- selected pool
- selected quote id
- selected return rate
- routing strategy
- decision reason

告警：

- Vendor disconnected
- Vendor quote stale
- no valid quote for active market
- quote reject spike
- route oscillation spike
- selected pool unhealthy
- order create failed after route selected

## 11. Rollout Plan

### Phase 0: Design confirmation

- 确认 inbound WS payload。
- 确认 Vendor Registry 存储方式。
- 确认 Phase 1 auth method。
- 确认 UAT 使用 immediate routing。

### Phase 1: Core implementation

- 新增 inbound vendor quote WS endpoint。
- 新增 Vendor Registry config/table。
- 新增 quote normalizer。
- 新增 quote book Redis schema。
- 新增 routing strategy interface 与 default best-quote implementation。
- 改造 `CreateOrderForPredictMarket` 使用 router 选择 `ReturnRate` 和 `PoolID`。
- 新增 order route audit。
- 改造 legacy SIG/SCB handler 写入统一 quote book。

### Phase 2: UAT

- 接入至少两个 Vendor source：
  - 一个 inbound push Vendor。
  - legacy SIG/SCB source。
- 验证 best quote selection。
- 验证 vendor-specific pool routing。
- 验证 stale quote reject。
- 验证 Vendor disconnect fallback。
- 验证 order open/settle 使用锁定 route。

### Phase 3: Production rollout

- Shadow 模式：接收 inbound quotes，只记录 hypothetical winner，不影响真实下单。
- Small traffic / allowlist：部分账户或市场启用 multi-vendor routing。
- Full traffic：所有配置市场启用。
- 观察 route oscillation 后决定是否启用 sticky/hysteresis。

### Phase 4: Legacy cleanup

- 关闭 legacy SIG/SCB outbound adapter。
- 移除 `PredictOtherPoolID` 的单一外部池硬编码语义。
- 保留 Vendor Registry + router 作为长期入口。

## 13. Open Questions

1. Inbound WebSocket auth method:
   - Recommended Phase 1: HMAC headers + optional network allowlist.
   - Need infra/security confirmation.
2. Quote win reaction speed:
   - UAT recommendation: immediate switching.
   - Production decision: immediate vs sticky window vs threshold/hysteresis.
3. Vendor Registry storage:
   - `sys_config` JSON for speed, or DB table for audit/admin.
4. Fallback behavior:
   - If no Vendor quote is valid, reject order by default.
   - Need business confirmation before enabling platform-pool fallback.
5. Capacity model:
   - Phase 1 can check pool health/basic balance.
   - Need future definition for per Vendor max exposure, per market notional cap, and side-specific capacity.
